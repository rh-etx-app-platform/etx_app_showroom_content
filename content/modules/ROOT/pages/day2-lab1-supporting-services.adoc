= Lab 1: Deploy Supporting Services
:navtitle: Supporting Services

== Overview

Deploy and configure the platform services required by the CI/CD pipeline and application observability: HashiCorp Vault for secrets management, OpenShift Pipelines (Tekton) for CI, and the OpenTelemetry Operator for distributed tracing.

== Deploy HashiCorp Vault

Vault provides centralized secrets management.
The pipeline will retrieve container registry credentials from Vault at runtime instead of storing them in Kubernetes Secrets.

=== Install Vault via Helm

Add the HashiCorp Helm repository and install Vault in standalone mode:

[source,bash]
----
helm repo add hashicorp https://helm.releases.hashicorp.com
helm repo update
----

Create a namespace and install:

[source,bash]
----
oc new-project vault

helm install vault hashicorp/vault \
  --namespace vault \
  --set "global.openshift=true" \
  --set "server.dev.enabled=true" \  # <1>
  --set "injector.enabled=false"
----
<1> Dev mode runs Vault unsealed with an in-memory backend. This is suitable for labs only -- never use dev mode in production.

Wait for the pod to be ready:

[source,bash]
----
oc get pods -n vault -w
----

Press Ctrl+C when the `vault-0` pod shows `1/1 Ready`.

=== Configure Vault

Exec into the Vault pod to configure secrets and authentication:

[source,bash]
----
oc exec -it vault-0 -n vault -- /bin/sh
----

Enable the KV v2 secrets engine and store registry credentials:

[source,bash]
----
vault secrets enable -path=secret kv-v2

# Store internal registry credentials (for dev environment)
vault kv put secret/registry \
  username=unused \
  password=unused \
  server=image-registry.openshift-image-registry.svc:5000

# Optionally, store Quay credentials (for stage/prod)
# vault kv put secret/quay \
#   username=<your_quay_username> \
#   password=<your_quay_password> \
#   server=quay-registry-quay.apps.cluster-cgd7r.dynamic.redhatworkshops.io
----

NOTE: The OpenShift internal registry uses ServiceAccount tokens for authentication, so username/password are not used. The Tekton pipeline will authenticate using its ServiceAccount automatically.

=== Configure Kubernetes Authentication

Enable the Kubernetes auth method so that pipeline ServiceAccounts can authenticate to Vault:

[source,bash]
----
vault auth enable kubernetes

vault write auth/kubernetes/config \
  kubernetes_host="https://$KUBERNETES_PORT_443_TCP_ADDR:443"

vault policy write pipeline-read - <<EOF
path "secret/data/registry" {
  capabilities = ["read"]
}
path "secret/data/quay" {
  capabilities = ["read"]
}
EOF

vault write auth/kubernetes/role/pipeline \
  bound_service_account_names=pipeline \
  bound_service_account_namespaces=parasol-dev \
  policies=pipeline-read \
  ttl=1h
----

Type `exit` to leave the Vault pod.

=== Verify Vault

Confirm you can read the secret from outside the pod:

[source,bash]
----
oc exec vault-0 -n vault -- vault kv get secret/registry
----

You should see the stored credentials.

== Install OpenShift Pipelines (Tekton)

OpenShift Pipelines provides the Tekton-based CI engine used to build, test, and push container images.

=== Install via OperatorHub

. In the OpenShift Web Console, go to menu:Operators[OperatorHub]
. Search for *Red Hat OpenShift Pipelines*
. Click btn:[Install]
. Accept the defaults:
  * **Update channel:** latest
  * **Installation mode:** All namespaces
  * **Installed Namespace:** openshift-operators
. Click btn:[Install]
. Wait for the operator to reach the `Succeeded` phase

Verify from the CLI:

[source,bash]
----
oc get csv -n openshift-operators | grep pipelines
----

You should see the Pipelines operator listed with phase `Succeeded`.

=== Verify Tekton Components

Confirm the Tekton controller pods are running:

[source,bash]
----
oc get pods -n openshift-pipelines
----

You should see pods for `tekton-pipelines-controller`, `tekton-triggers-controller`, and related components.

Verify the `tkn` CLI is available in your Dev Spaces workspace:

[source,bash]
----
tkn version
----

NOTE: The `tkn` CLI is included in the Universal Developer Image (UDI).

. Click btn:[Install]

  * **Installation mode:** All namespaces
  * **Installed Namespace:** openshift-operators
. Click btn:[Install]
. Wait for the operator to reach the `Succeeded` phase

[source,bash]
----
oc get csv -n openshift-operators | grep amq-streams

NOTE: The Kafka cluster will be deployed later as part of the application stack. The operator installation provides the CRDs (Kafka, KafkaTopic, etc.) needed to manage Kafka resources.

=== Deploy an OpenTelemetry Collector

Create a collector instance in your application namespace.
This collector will receive traces from instrumented applications and export them to the console for demonstration purposes:

[source,yaml]
----
apiVersion: opentelemetry.io/v1beta1
kind: OpenTelemetryCollector
metadata:
  name: otel-collector
  namespace: parasol-dev
spec:
  mode: deployment
  config:
    receivers:
      otlp:
        protocols:
          grpc:
            endpoint: 0.0.0.0:4317
          http:
            endpoint: 0.0.0.0:4318
    processors:
      batch: {}
    exporters:
      logging:
        loglevel: debug # <1>
    service:
      pipelines:
        traces:
          receivers: [otlp]
          processors: [batch]
          exporters: [logging] # <2>
----
<1> Set log level to debug to see trace details
<2> Export traces to stdout (visible in collector pod logs)

TIP: For production, replace the `logging` exporter with a real tracing backend like Tempo, Jaeger, or a cloud provider's tracing service.

Apply it:

[source,bash]
----
oc apply -f otel-collector.yaml
----

Verify the collector pod is running:

[source,bash]
----
oc get pods -l app.kubernetes.io/name=otel-collector-collector -n parasol-dev
----

=== Enable Auto-Instrumentation

Create an `Instrumentation` resource to enable automatic trace injection for Java applications:

[source,yaml]
----
apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: java-instrumentation
  namespace: parasol-dev
spec:
  exporter:
    endpoint: http://otel-collector-collector:4317
  propagators:
    - tracecontext
    - baggage
  java:
    image: ghcr.io/open-telemetry/opentelemetry-operator/autoinstrumentation-java:latest
----

Apply it:

[source,bash]
----
oc apply -f java-instrumentation.yaml
----

To instrument a deployment, add the following annotation to its pod template:

[source,yaml]
----
metadata:
  annotations:
    instrumentation.opentelemetry.io/inject-java: "true"
----

This will be used when deploying the application in xref:day2-lab1-app-deployment.adoc[Application Deployment].
