= Developer Hub: Templates and Integrations
:navtitle: Developer Hub: templates and integrations

Complete xref:day2-lab2-dh1-install-configure.adoc[Install and configure Red Hat Developer Hub] before this section so you have a running instance.

== Overview

In this section you will register a Golden Path or Software template that scaffolds new Quarkus projects with CI/CD
pre-configured, integrate Dev Spaces for one-click workspace creation, and configure OpenShift OAuth for enterprise authentication.

Software templates in Red Hat Developer Hub enable your team(s) to create Entities, such as new Components, and - through
the use of "actions" provided by plugins - create resources in other systems such as your GitLab, GitHub,
and OpenShift GitOps instances. Templates themselves are represented as Entities in the Software Catalog, meaning you can
import them similar to any other Entity!

Platform Engineers will often be the authors of Templates, and use them to create "golden paths" that follow
best-practices and use approved processes and tooling. Development teams will be the consumers of Templates to create new
software and automate their tasks. Using Templates reduces cognitive load on the development teams by allowing them to
focus on development tasks, while platform concerns are addressed by the template.

Templates are defined using YAML, but are rendered as a rich form in the Red Hat Developer Hub UI when used by development teams.

== Register a Golden Path Template

Golden Path templates provide a self-service way for developers to create new projects that follow organizational standards.

=== Template Structure

A Golden Path template consists of:

----
golden-path-quarkus/
├── template.yaml          # Template definition
└── skeleton/
    ├── devfile.yaml        # Dev Spaces workspace config
    ├── pom.xml             # Quarkus project with dependencies
    ├── src/                # Application source code
    ├── catalog-info.yaml   # Backstage catalog entity
    ├── tekton/             # Pipeline definitions
    │   ├── pipeline.yaml
    │   └── trigger.yaml
    └── gitops/             # Argo CD manifests
        ├── dev/
        │   ├── deployment.yaml
        │   ├── service.yaml
        │   └── route.yaml
        └── stage/
            ├── deployment.yaml
            ├── service.yaml
            └── route.yaml
----

=== Create the Template Definition

Create a Git repository in GitLab to hold the template definition.:

. Log into GitLab:
+
[source,text,subs="attributes+"]
----
{gitlab_url}
----
+
. Navigate to the Software Factory group
. Create a new empty and public project: *golden-path-quarkus* 
. Uncheck the "Initialize repository with a README" checkbox.
. Clone it locally:
+
[source,bash,role=execute,subs="attributes+"]
----
cd ~/workshop
git clone {gitlab_url}/software-factory/golden-path-quarkus.git
cd golden-path-quarkus
----

Create `template.yaml` in your Git repository:

[source,bash,role=execute,subs="attributes+"]
----
cat << 'EOF' > template.yaml
apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: quarkus-golden-path
  title: Quarkus Application (Golden Path)
  description: |
    Create a Quarkus application with CI/CD pipeline, GitOps deployment,
    Dev Spaces workspace, and OpenTelemetry instrumentation pre-configured.
  tags:
    - quarkus
    - java
    - recommended
spec:
  owner: platform-team
  type: service
  parameters:
    - title: Application Details
      required:
        - name
        - owner
      properties:
        name:
          title: Application Name
          type: string
          pattern: "^[a-z0-9-]+$"
        description:
          title: Description
          type: string
        owner:
          title: Owner
          type: string
          ui:field: OwnerPicker
        port:
          title: Port
          type: number
          default: 8080
          description: Override the port exposed for the application
        groupId:
          title: Group ID
          type: string
          default: io.janus
          description: Maven Group ID eg (io.janus)
        artifactId:
          title: Artifact ID
          type: string
          default: quarkusapp
          description: Maven Artifact ID
        javaPackageName:
          title: Java Package Namespace
          type: string
          default: io/janus/quarkusapp
          description: Name for the Java Package (ensure to use the / character as this is used for folder structure) should match Group ID and Artifact ID

    - title: Repository
      required:
        - repoUrl
      properties:
        repoUrl:
          title: Repository Location
          type: string
          ui:field: RepoUrlPicker
          ui:options:
            allowedHosts:
              - gitlab.{openshift_cluster_ingress_domain}
            allowedOwners:
              - software-factory
  steps:
    - id: fetch-skeleton
      name: Fetch Skeleton
      action: fetch:template
      input:
        url: ./skeleton
        values:
          name: ${{ parameters.name }}
          description: ${{ parameters.description }}
          owner: ${{ parameters.owner }}
          applicationType: api
          port: ${{ parameters.port }}
          groupId: ${{ parameters.groupId }}
          artifactId: ${{ parameters.artifactId }}
          javaPackageName: ${{ parameters.javaPackageName }}

    - id: publish
      name: Publish to GitLab
      action: publish:gitlab
      input:
        repoUrl: ${{ parameters.repoUrl }}
        settings:
          description: ${{ parameters.description }}
        defaultBranch: main
        repoVisibility: public

    - id: register
      name: Register in Catalog
      action: catalog:register
      input:
        repoContentsUrl: ${{ steps['publish'].output.repoContentsUrl }}
        catalogInfoPath: /catalog-info.yaml

  output:
    links:
      - title: Repository
        url: ${{ steps['publish'].output.remoteUrl }}
      - title: Open in Catalog
        icon: catalog
        entityRef: ${{ steps['register'].output.entityRef }}
EOF
----

=== Create the Skeleton

The skeleton is a template application, in this case Quarkus this is used to give the developer a head start.  It is intended to allow for setting up company standards, etc.  It could include CI/CD pipelines as well but that is outside the scope of this lab and will be showcased in the TSSC labs later.

Run the following to create a sample skeleton:

[source,bash,role=execute]
----
curl -L \
  "https://codeload.github.com/redhat-gpte-devopsautomation/software-templates/tar.gz/main" \
  | tar -xz --strip-components=3 \
    software-templates-main/scaffolder-templates/quarkus-backend-template/skeleton
----

=== Create the Catalog Entity

Add `catalog-info.yaml` to the skeleton directory:

[source,bash,role=execute]
----
cat << 'EOF' > skeleton/catalog-info.yaml
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: ${{ values.name }}
  description: ${{ values.description }}
  annotations:
    backstage.io/techdocs-ref: dir:.
    argocd/app-name: ${{ values.name }}-dev
    janus-idp.io/tekton: ${{ values.name }}
spec:
  type: service
  lifecycle: production
  owner: ${{ values.owner }}
EOF
----

Commit and push the template repository:

[source,bash,role=execute]
----
git add .
git commit -m "feat: initial commit"
git push origin main
----

=== Register the Template in Developer Hub

Red Hat Developer Hub discovers Software Templates through *catalog locations* in `app-config.yaml`.
After you push `template.yaml` to GitLab, add a `catalog` section that points at that file so the catalog importer
loads your Golden Path template.
You update the existing `rhdh-config` ConfigMap in the `setup-rhdh` namespace—the same ConfigMap created in xref:day2-lab2-dh1-install-configure.adoc[Install and configure Red Hat Developer Hub].

. Open {openshift_console_url}[^] and switch to the *setup-rhdh* project.
. In the console, open **Workloads** → **ConfigMaps** and select *rhdh-config*.
. Click **Actions** → *Edit ConfigMap* and stay in **YAML view** (or open the **Data** tab and edit the `app-config.yaml` key).
. Under `data.app-config.yaml`, keep your existing `app` and `auth` settings and add the `catalog` block below.
  Do not replace the whole file—merge the new section with what is already there.
+
[source,yaml,subs="attributes+"]
----
    catalog:
      locations:
        - type: url
          target: '{gitlab_url}/software-factory/golden-path-quarkus/-/raw/main/template.yaml'
          rules:
            - allow: [Template]
----
+
. Click **Save** to apply the ConfigMap.

The Red Hat Developer Hub Operator rolls out the change and restarts the application pod.
When the pod is `Running` again, open Developer Hub and confirm the template appears under menu:Create[Templates].

=== Setup GitLab Integration

Create a Person Access Token from GitLab to be used in RHDH.  In production you would use a service account but for the purposes of this lab we can use a PAT.

. Log into GitLab:
+
[source,text,subs="attributes+"]
----
{gitlab_url}
----
+
. Navigate to your User settings
. Click the Access - Personal access tokens 
. Click `Add new token`
. Give it a name and select the `api` scope and click Generate Token
. You will see a Green box that says `Your token`, copy that token and run the following replacing `{YOUR_PAT_HERE}` with the PAT you copied:
+
[source,bash,role=execute]
----
export GITLAB_PAT={YOUR_PAT_HERE}
----

NOTE: We will use the `GITLAB_PATH` environment variable in a later step when we setup the RHDH auth secret

== Integrate with Dev Spaces

Add a Dev Spaces link to catalog components so developers can open a workspace directly from the component page.

=== Add the Dev Spaces URL Annotation

In the `catalog-info.yaml` skeleton, add the Dev Spaces annotation:

[source,yaml,subs="attributes+"]
----
metadata:
  annotations:
    che.eclipse.org/che-server: {devspaces_url}
----

WARNING: Do not forget to commit and push the changes to the Git repository.

[source,bash,role=execute]
----
git add catalog-info.yaml
git commit -m "feat: add dev spaces annotation"
git push origin main
----

When a component has this annotation, the Developer Hub UI shows a "Dev Spaces" link that opens a workspace for the component's repository.

=== Configure the Dev Spaces Plugin

Add the Dev Spaces plugin configuration to *rhdh-config* ConfigMap. Under `data.app-config.yaml`, keep your
existing `app`, `auth` and other settings and add the `devSpaces` block below.
Do not replace the whole file—merge the new section with what is already there.

[source,yaml,subs="attributes+"]
----
    devSpaces:
      defaultUrl: {devspaces_url}
----

Update the ConfigMap and wait for the pod to restart.

== Configure Keycloak OAuth Authentication

Replace guest authentication with Keycloak OAuth for production-like access.

=== Create the Auth Secret

RHDH will need to sign/encrypt the session cookie used during the OIDC flow.  We need to generate and define a session secret.  It can be any string but for real use it should be high entropy, e.g. ~32 random bytes

[source,bash,role=execute,subs="attributes+"]
----
export SESSION_SECRET=$(python3 -c 'import secrets; print(secrets.token_hex(32))')

oc create secret generic rhdh-auth-secret -n setup-rhdh \
  --from-literal=KEYCLOAK_REALM=etx-lab \
  --from-literal=KEYCLOAK_CLIENT_ID=rhdh \
  --from-literal=KEYCLOAK_CLIENT_SECRET=etx-lab-workshop-gitlab-secret \
  --from-literal=KEYCLOAK_BASE_URL=https://etx-sso.{openshift_cluster_ingress_domain} \
  --from-literal=SESSION_SECRET=${SESSION_SECRET} \
  --from-literal=GITLAB_TOKEN=${GITLAB_PAT}
----

=== Update app-config

* Replace the guest auth block with OIDC and the session secret from above
* Update the catalog provide to sync the Keycloak users. 
* Include the gitlab integration

WARNING: Do not replace the catalog key or you will overwrite your template location from before.  Just add the `providers` section

[source,yaml]
----
    signInPage: oidc
    auth:
      environment: production
      session:
        secret: ${SESSION_SECRET}
      providers:
        oidc:
          production:
            clientId: ${KEYCLOAK_CLIENT_ID}
            clientSecret: ${KEYCLOAK_CLIENT_SECRET}
            metadataUrl: ${KEYCLOAK_BASE_URL}/realms/${KEYCLOAK_REALM}/.well-known/openid-configuration
            prompt: auto
    catalog:
      providers:
        keycloakOrg:
          default:
            baseUrl: ${KEYCLOAK_BASE_URL}
            clientId: ${KEYCLOAK_CLIENT_ID}
            clientSecret: ${KEYCLOAK_CLIENT_SECRET}
            realm: ${KEYCLOAK_REALM}
            loginRealm: ${KEYCLOAK_REALM}
            schedule:
              frequency: { minutes: 5 }
    integrations:
      gitlab:
        - host: gitlab.{openshift_cluster_ingress_domain}
          token: ${GITLAB_TOKEN}
          apiBaseUrl: {gitlab_url}/api/v4
          baseUrl: {gitlab_url}
----

NOTE: The schedule frequency would be set to something higher in production (12 hours) and is set low for demo purposes.  Other options like `userQuerySize` or `groupQuerySize` are also good to set.  More information found in the documentation https://docs.redhat.com/en/documentation/red_hat_developer_hub/1.10/html/authentication_in_red_hat_developer_hub/import-users-and-groups-from-your-identity-provider_authentication-in-rhdh#import-users-and-groups-from-rhbk_import-users-and-groups-from-your-identity-provider[here]


Click **Save** and wait for the Backstage pod to restart.

`signInPage: oidc` shows the Keycloak sign-in flow on the login page.
The `${KEYCLOAK_*}` placeholders are resolved from the Secret mounted on the `Backstage` CR in the next step.

=== Create Dynamic Plugins ConfigMap
Create a dynamic plugins ConfigMap to enable the Keycloak Org sync plugin and the gitlab scaffolder plugin for our template

[source,bash,role=execute,subs="attributes+"]
----
oc create -f - <<EOF
kind: ConfigMap
apiVersion: v1
metadata:
  name: dynamic-plugins-rhdh
  namespace: setup-rhdh
data:
  dynamic-plugins.yaml: |
    includes:
      - dynamic-plugins.default.yaml
    plugins:
      - package: './dynamic-plugins/dist/backstage-community-plugin-catalog-backend-module-keycloak-dynamic'
        disabled: false
      - package: './dynamic-plugins/dist/backstage-plugin-scaffolder-backend-module-gitlab-dynamic'
        disabled: false
EOF
----

=== Update the Backstage CR

* Mount `rhdh-auth-secret` on the `Backstage` custom resource so the OIDC client ID, secret, and metadata URL are available to `app-config` as environment variables.
* Add the dynamic-plugins-rhdh ConfigMap so we can enable the keycloak Org sync

. Open {openshift_console_url}[^] and switch to the *setup-rhdh* project.
. Go to **Ecosystem** → **Installed Operators**.
. Select *Red Hat Developer Hub Operator*.
. Open the **Red Hat Developer Hub** tab.
. Click the *rhdh* instance created.
. Click **Actions** → *Edit Backstage* and switch to **YAML view**.
. Under `spec.application`, add the `extraEnvs` & `dynamicPluginsConfigMapName` blocks below.
  Keep the existing `appConfig` and `route` sections from the initial deployment.
+
[source,yaml]
----
    extraEnvs:
      secrets:
        - name: rhdh-auth-secret
    dynamicPluginsConfigMapName: dynamic-plugins-rhdh
----
+
. Click **Save**.

The Red Hat Developer Hub Operator reconciles the CR and restarts the Backstage pod.

=== Verify Keycloak sign-in

. Wait until the Backstage pod in *setup-rhdh* is `Running`.
. Open the Developer Hub route URL from the first step in this section.
. On the login page, choose *OIDC*.
. Sign in with your Keycloak credentials (for example `{user}` / `{password}`).
+
You should land on the Developer Hub home page authenticated as your Keycloak user, not as Guest.

IMPORTANT: The signed-in user must exist in the Software Catalog (or be mapped via your org setup) to use some catalog features. For template creation in this lab, guest access is no longer available after this change.

== Test the Golden Path

. Open Developer Hub and go to menu:Create[Templates]
. Select the *Quarkus Application (Golden Path)* template
. Fill in the parameters (name, owner, repository location)
. Click btn:[Create]
. Verify the template:
  * A new GitLab repository is created with the skeleton code
  * The component is registered in the Developer Hub catalog
  * The repository includes a devfile, Tekton pipeline, and GitOps manifests
. Open the component in the catalog and click the Dev Spaces link to launch a workspace

== Summary

Developer Hub now provides:

* *Golden Path templates* that scaffold complete projects with CI/CD, GitOps, and Dev Spaces pre-configured
* *Software Catalog* with component metadata, ownership, and links to pipelines and deployments
* *Dev Spaces integration* for one-click workspace creation from catalog components
* *OpenShift OAuth* for enterprise authentication
