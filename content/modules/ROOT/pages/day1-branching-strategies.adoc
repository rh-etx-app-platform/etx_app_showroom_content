== Branching Models

Different teams use different branching strategies based on their release cadence, team size, and deployment practices.

=== Trunk-Based Development

image::https://trunkbaseddevelopment.com/trunk1b.png[Trunk-Based Development,400]

**Characteristics:**

* All developers commit to a single `main` branch
* Short-lived feature branches (hours to days)
* Frequent integration (multiple times per day)
* Continuous Integration runs on every commit
* Feature flags for incomplete features

**Best suited for:**

* Teams practicing Continuous Delivery
* Small to medium teams (2-10 developers)
* Projects with automated testing
* Cloud-native applications

**Example workflow:**

[source,bash]
----
# Create short-lived feature branch
git checkout -b feature/add-validation

# Work, commit frequently
git commit -am "add email validation"

# Merge back quickly (same day)
git checkout main
git merge feature/add-validation
git push origin main
----

=== GitFlow

image::https://nvie.com/img/git-model@2x.png[GitFlow,400]

**Characteristics:**

* Multiple long-lived branches: `main`, `develop`, `release/*`, `hotfix/*`
* Feature branches created from `develop`
* Release branches for production preparation
* Hotfix branches for urgent production fixes
* Well-defined promotion path

**Best suited for:**

* Teams with scheduled release cycles (monthly, quarterly)
* Larger teams with dedicated release managers
* Projects requiring multiple supported versions
* Enterprise applications with formal change control

**Branch structure:**

* `main` - Production-ready code
* `develop` - Integration branch for features
* `feature/*` - New features (from develop)
* `release/*` - Preparing next production release
* `hotfix/*` - Emergency production fixes (from main)

=== GitHub Flow

image::https://miro.medium.com/v2/resize:fit:720/format:webp/1*S4VJPf8Mrg3Gn5cR7JK62w.png[GitHub Flow,400]

**Characteristics:**

* Simplified model with `main` + feature branches
* Pull Requests for all changes
* CI/CD runs on every PR
* Deploy from `main` after merge
* No separate develop branch

**Best suited for:**

* Teams practicing Continuous Deployment
* Web applications deployed frequently
* Projects with strong automated testing
* Open source projects

**Example workflow:**

[source,bash,subs="attributes"]
----
# Create feature branch
git checkout -b feature/user-profile

# Work and push frequently
git commit -am "add user profile page"
git push origin feature/user-profile

# Open Pull Request in GitLab
# CI runs automatically
# Deploy to staging for review
# Merge to main after approval
# Deploy to production automatically
----

=== Choosing the Right Model

[cols="1,2,2,2",options="header"]
|===
|Model |Release Frequency |Team Size |Complexity

|Trunk-Based
|Continuous
|Small-Medium
|Low

|GitFlow
|Scheduled
|Medium-Large
|High

|GitHub Flow
|Continuous
|Any
|Medium
|===

**For this workshop**, we use **Trunk-Based Development** concepts:

* Short-lived feature branches
* Frequent integration
* CI/CD automation
* GitOps for deployment

== Lab Exercise: Branching in Practice

Now let's practice the fundamental branching operations you'll use throughout Day 2 labs.

=== Exercise 1: Create a Feature Branch

Create a new branch for your changes:

[source,bash,role=execute]
----
# Ensure you're on main
git checkout main

# Create and switch to feature branch
git checkout -b feature/update-readme

# Verify you're on the new branch
git branch
----

.Expected Output
[source,text]
----
* feature/update-readme
  main
----

The asterisk (*) indicates your current branch.

[NOTE]
====
**Branch naming convention:**

* `feature/` - New features
* `fix/` - Bug fixes
* `docs/` - Documentation changes
* `refactor/` - Code refactoring

Example: `feature/add-authentication`, `fix/login-bug`
====

=== Exercise 2: Make Changes and Commit

Make a simple change to the README:

[source,bash,role=execute]
----
# Add your name to the README
cat >> README.md << 'EOF'

## Workshop Participant
- Name: Your Name
- Date: $(date +%Y-%m-%d)
EOF

# View what changed
git status
git diff README.md
----

Stage and commit your changes:

[source,bash,role=execute]
----
# Stage the file
git add README.md

# Commit with a descriptive message
git commit -m "docs: add workshop participant information"

# View your commit
git log --oneline -1
----

.Commit Message Format
[TIP]
====
Use conventional commit format:

* `feat:` - New feature
* `fix:` - Bug fix
* `docs:` - Documentation
* `refactor:` - Code restructuring
* `test:` - Adding tests

Example: `feat: add user authentication`
====

=== Exercise 3: Push to Remote

Push your branch to GitLab:

[source,bash,role=execute]
----
# Push to remote (first time for this branch)
git push -u origin feature/update-readme
----

=== Exercise 5: Merge Your Changes

In a real workflow, you'd create a Merge Request for review. For this exercise, merge directly:

[source,bash,role=execute]
----
# Switch to main
git checkout main

# Merge your feature branch
git merge feature/update-readme
----

.Expected Output (Fast-forward merge)
[source,text]
----
Updating a1b2c3d..e4f5g6h
Fast-forward
 README.md | 4 ++++
 1 file changed, 4 insertions(+)
----

[NOTE]
====
**Fast-forward merge:** Git simply moves the `main` pointer forward because no other changes were made to `main`. This is the simplest type of merge.
====

Push the merged changes:

[source,bash,role=execute]
----
git push origin main
----

Clean up the feature branch:

[source,bash,role=execute]
----
# Delete local branch
git branch -d feature/update-readme

# Delete remote branch
git push origin --delete feature/update-readme
----

== Key Takeaways

**Branching enables parallel development:**

* Multiple developers work on different features simultaneously
* Isolated changes until ready for integration
* Code review through merge requests
* CI/CD automation on every push

**Different models suit different needs:**

* Choose based on release cadence, team size, and deployment frequency
* This workshop uses Trunk-Based Development principles
* Day 2 labs will integrate branches with CI/CD pipelines

**Best practices:**

* Keep branches short-lived (hours to days)
* Use descriptive branch names and commit messages
* Push frequently to back up your work
* Delete branches after merging
