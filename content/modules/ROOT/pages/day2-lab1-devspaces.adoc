= Lab 1: Dev Spaces and IDE Setup
:navtitle: Dev Spaces & IDE Setup

== Overview

In this lab you will create a Dev Spaces workspace for the ETX App Base application, verify Git and AI assistant integration, and run Quarkus Dev Services with nested containers.

== Create a Devfile

Create a new `devfile.yaml` file in the root of your cloned repository. This devfile configures the workspace with:

* Universal Developer Image (UBI) with Java tooling
* Maven caching for faster builds
* Quarkus development endpoints
* Continue AI assistant integration
* **Podman/Testcontainers configuration** - Required for Quarkus Dev Services to work in Dev Spaces

IMPORTANT: The repository does NOT include a pre-existing devfile. You must create this file manually following these instructions.

In the root directory of your cloned `etx_app_base_app` repository, create a new file named `devfile.yaml` with the following content:

[source,yaml]
----
schemaVersion: 2.2.0
metadata:
  name: etx-app-base-app
components:
  - name: development-tooling
    container:
      image: quay.io/devfile/universal-developer-image:ubi9-latest
      env:
        - name: QUARKUS_HTTP_HOST
          value: "0.0.0.0"
        - name: MAVEN_OPTS
          value: "-Dmaven.repo.local=/home/user/.m2/repository"
        # Critical: Testcontainers/Quarkus Dev Services configuration for Podman
        - name: XDG_RUNTIME_DIR # <1>
          value: "/tmp/podman-run-1000"
        - name: DOCKER_HOST # <2>
          value: "unix:///tmp/podman-run-1000/podman/podman.sock"
        - name: TESTCONTAINERS_RYUK_DISABLED # <3>
          value: "true"
        - name: TESTCONTAINERS_DOCKER_SOCKET_OVERRIDE # <4>
          value: "/tmp/podman-run-1000/podman/podman.sock"
      memoryLimit: 5Gi
      cpuLimit: 2500m
      volumeMounts:
        - name: m2
          path: /home/user/.m2
      endpoints:
        - name: quarkus-dev
          targetPort: 8080
          exposure: public
          protocol: https
        - name: debug
          targetPort: 5005
          exposure: none
  - name: m2
    volume:
      size: 1G
commands:
  - id: package
    exec:
      label: "1. Package the application"
      component: development-tooling
      commandLine: "./mvnw package"
      group:
        kind: build
        isDefault: true
  - id: start-dev
    exec:
      label: "2. Start Development mode (Hot reload + debug)"
      component: development-tooling
      commandLine: "./mvnw compile quarkus:dev"
      group:
        kind: run
        isDefault: true
  - id: init-podman
    exec:
      label: "Initialize Podman socket"
      component: development-tooling
      commandLine: |
        #!/bin/bash
        mkdir -p ${XDG_RUNTIME_DIR}/podman
        nohup podman system service --time=0 unix://${XDG_RUNTIME_DIR}/podman/podman.sock > /tmp/podman.log 2>&1 &
        echo "Waiting for Podman socket to be ready..."
        for i in {1..30}; do
          if [ -S ${XDG_RUNTIME_DIR}/podman/podman.sock ]; then
            echo "Podman socket is ready at ${XDG_RUNTIME_DIR}/podman/podman.sock"
            exit 0
          fi
          sleep 1
        done
        echo "ERROR: Podman socket failed to start"
        exit 1
      group:
        kind: build
  - id: init-continue
    exec:
      label: "Initialize Continue config"
      component: development-tooling
      workingDir: /home/user
      commandLine: |
        mkdir -p /home/user/.continue
        cat > /home/user/.continue/config.yaml << 'INNEREOF'
        name: Continue Config
        version: 0.0.1
        models:
          - name: qwen3-14b
            provider: vllm
            model: qwen3-14b
            apiKey: ${LLM_API_KEY}
            apiBase: ${LLM_BASE_URL}
            roles:
              - chat
        INNEREOF
      group:
        kind: build
events:
  postStart:
    - init-podman
    - init-continue
----
<1> XDG_RUNTIME_DIR is required for Podman to know where to create its runtime socket. Without this, the socket won't be created.
<2> Points Testcontainers to use Podman socket instead of Docker. Without this, you'll get: `IllegalStateException: Previous attempts to find a Docker environment failed`
<3> Disables the Ryuk cleanup container which is not compatible with Podman in Dev Spaces
<4> Overrides the socket path for Testcontainers compatibility with Podman

IMPORTANT: These four environment variables (`XDG_RUNTIME_DIR`, `DOCKER_HOST`, `TESTCONTAINERS_RYUK_DISABLED`, `TESTCONTAINERS_DOCKER_SOCKET_OVERRIDE`) plus the `init-podman` postStart command are **critical** for Quarkus Dev Services to work in Dev Spaces. Without them, Maven builds will fail with "Docker not found" errors when trying to start test containers for Kafka and PostgreSQL.

After creating the file, commit and push the devfile so Dev Spaces can read it when creating your workspace:

[source,bash]
----
git add devfile.yaml
git commit -m "feat: add devfile with Continue AI assistant"
git push
----

== Get the Dashboard URL

[source,bash]
----
oc get checluster/devspaces -n openshift-devspaces \
  -o jsonpath='{.status.cheURL}'
----

Open this URL in your browser and log in with your OpenShift credentials.

== Create the Lab Workspace

The simplest way to create a workspace is by URL.
Open this URL to create a workspace from the ETX App Base application:

[source,bash,role=execute,subs="attributes"]
----
{devspaces_url}/#{gitlab_url}/software-factory/etx_app_base_app/-/tree/main?devfilePath=/
----

To create a workspace from a different branch, replace `REPLACE_BRANCH_NAME` in this URL:

[source,bash,role=execute,subs="attributes"]
----
{devspaces_url}/#{gitlab_url}/software-factory/etx_app_base_app/-/tree/REPLACE_BRANCH_NAME?devfilePath=/
----

Alternatively, from the Dev Spaces dashboard:

. Click btn:[Create Workspace]
. Go to Import from Git and paste the Git repository URL for the ETX App Base application
. Click btn:[Create & Open]

Dev Spaces will clone the repository, read the `devfile.yaml` you just created, pull the container image, and open the VS Code IDE in your browser.

NOTE: The first workspace creation may take a few minutes while the container image is pulled to the node.

== Verify the AI Code Assistant

Open a terminal in the workspace and confirm the LLM environment variables are set:

[source,bash]
----
echo $LLM_API_KEY
echo $LLM_BASE_URL
----

Both should return values injected by the shared secret configured in the xref:day2-devspaces-integrations.adoc[Shared Secrets] section.

Check that the Continue extension is configured:

. Open the Continue panel in the sidebar (the Continue icon)
. Send a test message such as "Hello, what model are you?"
. Confirm you receive a response from the **qwen3-14b** model

[source,bash]
----
cat ~/.continue/config.yaml
----

The file should contain the model configuration with your API key and base URL.

== Verify Nested Containers

Confirm Podman is available inside the workspace:

[source,bash]
----
podman version
podman run --rm registry.access.redhat.com/ubi9-minimal:latest echo "Nested containers work"
----

Starting with Dev Spaces 3.25 on OpenShift 4.20+, nested containers work without special pod annotations or SCC configuration.
See the https://docs.redhat.com/en/documentation/red_hat_openshift_dev_spaces/3.25/html/release_notes_and_known_issues/new-features[Dev Spaces 3.25 Release Notes^] for details.

When `disableContainerRunCapabilities: false` is set in the CheCluster CR, workspace pods get the capabilities needed to run Podman in rootless mode.
Quarkus Dev Services uses Testcontainers under the hood, which calls Podman to start service containers (databases, message brokers, etc.).

=== Configure Podman Socket for Testcontainers

Testcontainers expects to communicate with a Docker daemon, but in DevSpaces we use Podman. The devfile includes environment variables that configure Testcontainers to use Podman:

* `DOCKER_HOST` - Points to the Podman socket instead of Docker
* `TESTCONTAINERS_RYUK_DISABLED` - Disables the Ryuk cleanup container (not compatible with Podman in DevSpaces)
* `TESTCONTAINERS_DOCKER_SOCKET_OVERRIDE` - Overrides the socket path for compatibility

Verify the Podman socket is running:

[source,bash]
----
podman info --format '{{.Host.RemoteSocket.Path}}'
----

This should return `/tmp/podman-run-1000/podman/podman.sock`.

Verify the socket file exists and is accessible:

[source,bash]
----
ls -la ${XDG_RUNTIME_DIR}/podman/podman.sock
----

You should see a socket file (type `srw-------`).

**Troubleshooting:** If you encounter this error when running `./mvnw compile quarkus:dev`:

[source]
----
java.lang.IllegalStateException: Previous attempts to find a Docker environment failed.
Will not retry. Please see logs and check configuration
----

Or this warning:

[source]
----
Unix socket defined in DOCKER_HOST is not writable, make sure Docker is running
----

This means either:

1. The Podman socket service didn't start automatically, or
2. Your workspace was created before the devfile was updated

**Solution 1: Recreate the workspace** (recommended):

Delete your current workspace from the Dev Spaces dashboard and create a new one using the updated devfile. The `init-podman` postStart command will automatically start the Podman socket.

**Solution 2: Manual fix for current workspace**:

Start the Podman socket manually and set the environment variables:

[source,bash]
----
# Start Podman socket
export XDG_RUNTIME_DIR=/tmp/podman-run-1000
mkdir -p ${XDG_RUNTIME_DIR}/podman
nohup podman system service --time=0 unix://${XDG_RUNTIME_DIR}/podman/podman.sock > /tmp/podman.log 2>&1 &

# Wait for socket to be ready
sleep 3
ls -la ${XDG_RUNTIME_DIR}/podman/podman.sock

# Set Testcontainers variables
export DOCKER_HOST=unix:///tmp/podman-run-1000/podman/podman.sock
export TESTCONTAINERS_RYUK_DISABLED=true
export TESTCONTAINERS_DOCKER_SOCKET_OVERRIDE=/tmp/podman-run-1000/podman/podman.sock

# Verify configuration
echo "XDG_RUNTIME_DIR: $XDG_RUNTIME_DIR"
echo "DOCKER_HOST: $DOCKER_HOST"
podman info --format '{{.Host.RemoteSocket.Path}}'
----

== Run Quarkus Dev Services

Open a terminal in the IDE and start Quarkus in dev mode:

[source,bash]
----
./mvnw compile quarkus:dev
----

Quarkus Dev Services will detect that the application needs a PostgreSQL database and automatically start one using Podman inside the workspace.
You should see log output indicating the database container is starting:

----
Dev Services for default datasource (postgresql) started
----

The application will be available on port 8080.
Click the endpoint notification in the IDE or find the URL in the *Endpoints* panel.

== Workspace Lifecycle

Some useful operations:

*Stop a workspace* (preserves state on the PVC):

* From the dashboard, click the stop icon next to your workspace
* The workspace pod is deleted, but the PVC retains your files

*Restart a workspace*:

* Click the workspace name in the dashboard to restart it
* Your source code, Maven cache, and other volume-mounted data persist

*Delete a workspace*:

* From the dashboard, select the workspace and click btn:[Delete]
* This removes the workspace pod and its PVC
