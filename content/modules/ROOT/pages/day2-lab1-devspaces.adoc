= Lab 1: Dev Spaces and IDE Setup
:navtitle: Dev Spaces & IDE Setup

== Overview

In this lab you will create a Dev Spaces workspace for the ETX App Base application, verify Git and AI assistant integration, and run Quarkus Dev Services with nested containers.

== Create a Devfile

Create a new `devfile.yaml` file in the root of your cloned repository. This devfile configures the workspace with:

* Universal Developer Image (UBI) with Java tooling
* Maven caching for faster builds
* Quarkus development endpoints
* Continue AI assistant integration
* **Environment variables** - Connection details for external PostgreSQL and Kafka services

IMPORTANT: The repository does NOT include a pre-existing devfile. You must create this file manually following these instructions.

In the root directory of your cloned `etx_app_base_app` repository, create a new file named `devfile.yaml` with the following content:

[source,yaml]
----
schemaVersion: 2.2.0
metadata:
  name: etx-app-base-app
components:
  - name: development-tooling
    container:
      image: quay.io/devfile/universal-developer-image:ubi9-latest
      env:
        - name: QUARKUS_HTTP_HOST
          value: "0.0.0.0"
        - name: MAVEN_OPTS
          value: "-Dmaven.repo.local=/home/user/.m2/repository"
        # External service connection (from Supporting Services lab)
        - name: POSTGRESQL_HOST # <1>
          value: "parasol-db.etx-app-dev.svc.cluster.local"
        - name: POSTGRESQL_DATABASE # <2>
          value: "parasol"
        - name: POSTGRESQL_USER # <3>
          value: "parasol"
        - name: POSTGRESQL_PASSWORD # <4>
          value: "parasol"
        - name: KAFKA_BOOTSTRAP_SERVERS # <5>
          value: "parasol-kafka-kafka-bootstrap.etx-app-dev.svc.cluster.local:9092"
      memoryLimit: 5Gi
      cpuLimit: 2500m
      volumeMounts:
        - name: m2
          path: /home/user/.m2
      endpoints:
        - name: quarkus-dev
          targetPort: 8080
          exposure: public
          protocol: https
        - name: debug
          targetPort: 5005
          exposure: none
  - name: m2
    volume:
      size: 1G
commands:
  - id: package
    exec:
      label: "1. Package the application"
      component: development-tooling
      commandLine: "./mvnw package"
      group:
        kind: build
        isDefault: true
  - id: start-dev
    exec:
      label: "2. Start Development mode (Hot reload + debug)"
      component: development-tooling
      commandLine: "./mvnw compile quarkus:dev"
      group:
        kind: run
        isDefault: true
  - id: init-continue
    exec:
      label: "Initialize Continue config"
      component: development-tooling
      workingDir: /home/user
      commandLine: |
        mkdir -p /home/user/.continue
        cat > /home/user/.continue/config.yaml << 'INNEREOF'
        name: Continue Config
        version: 0.0.1
        models:
          - name: qwen3-14b
            provider: vllm
            model: qwen3-14b
            apiKey: ${LLM_API_KEY}
            apiBase: ${LLM_BASE_URL}
            roles:
              - chat
        INNEREOF
      group:
        kind: build
events:
  postStart:
    - init-continue
----
<1> PostgreSQL hostname - fully qualified service name from Supporting Services lab
<2> PostgreSQL database name (matches the database created in Supporting Services)
<3> PostgreSQL username (matches the secret in Supporting Services)
<4> PostgreSQL password (matches the secret in Supporting Services)
<5> Kafka bootstrap servers - fully qualified service name from Supporting Services lab

IMPORTANT: These environment variables point to the external PostgreSQL and Kafka services you deployed in xref:day2-lab1-supporting-services.adoc[Deploy Supporting Services]. Your application will connect to these shared services instead of starting local containers.

After creating the file, commit and push the devfile so Dev Spaces can read it when creating your workspace:

[source,bash]
----
git add devfile.yaml
git commit -m "feat: add devfile with Continue AI assistant"
git push
----

See xref:day2-devspaces-devfiles.adoc[Devfiles & Base Images] for a full explanation of each section.

== Get the Dashboard URL

[source,bash]
----
oc get checluster/devspaces -n openshift-devspaces \
  -o jsonpath='{.status.cheURL}'
----

Open this URL in your browser and log in with your OpenShift credentials.

== Create the Lab Workspace

The simplest way to create a workspace is by URL.
Open this URL to create a workspace from the ETX App Base application:

[source,bash,role=execute,subs="attributes"]
----
{devspaces_url}/#{gitlab_url}/software-factory/etx_app_base_app/-/tree/main?devfilePath=/
----

To create a workspace from a different branch, replace `REPLACE_BRANCH_NAME` in this URL:

[source,bash,role=execute,subs="attributes"]
----
{devspaces_url}/#{gitlab_url}/software-factory/etx_app_base_app/-/tree/REPLACE_BRANCH_NAME?devfilePath=/
----

Alternatively, from the Dev Spaces dashboard:

. Click btn:[Create Workspace]
. Go to Import from Git and paste the Git repository URL for the ETX App Base application
. Click btn:[Create & Open]

Dev Spaces will clone the repository, read the `devfile.yaml` you just created, pull the container image, and open the VS Code IDE in your browser.

NOTE: The first workspace creation may take a few minutes while the container image is pulled to the node.

== Verify Git Integration

Once the workspace opens:

. Open the *Source Control* panel (Ctrl+Shift+G)
. Confirm the repository is loaded and shows the correct branch
. If prompted to authenticate with GitLab, follow the OAuth flow -- this is the integration configured in the xref:day2-devspaces-integrations.adoc[GitLab Integration] section

Test that you can push changes:

. Create or edit a file (e.g. add a comment to `README.md`)
. Stage, commit, and push the change from the Source Control panel
. Verify the commit appears in GitLab

== Verify the AI Code Assistant

Open a terminal in the workspace and confirm the LLM environment variables are set:

[source,bash]
----
echo $LLM_API_KEY
echo $LLM_BASE_URL
----

Both should return values injected by the shared secret configured in the xref:day2-devspaces-integrations.adoc[Shared Secrets] section.

Check that the Continue extension is configured:

. Open the Continue panel in the sidebar (the Continue icon)
. Send a test message such as "Hello, what model are you?"
. Confirm you receive a response from the **qwen3-14b** model

If Continue does not appear, check that the `postStart` event ran successfully:

[source,bash]
----
cat ~/.continue/config.yaml
----

The file should contain the model configuration with your API key and base URL.

== Verify Service Connection

Verify that the workspace can reach the external services deployed in the previous lab:

Test PostgreSQL connection:

[source,bash]
----
echo $POSTGRESQL_HOST
echo $POSTGRESQL_DATABASE
# Should show: parasol-db.etx-app-dev.svc.cluster.local and parasol
----

Test Kafka connection:

[source,bash]
----
echo $KAFKA_BOOTSTRAP_SERVERS
# Should show: parasol-kafka-kafka-bootstrap.etx-app-dev.svc.cluster.local:9092
----

Verify network connectivity to PostgreSQL:

[source,bash]
----
nc -zv $POSTGRESQL_HOST 5432
----

You should see `Connection to parasol-db.etx-app-dev.svc.cluster.local 5432 port [tcp/postgresql] succeeded!`

IMPORTANT: If you get connection errors, verify that you completed xref:day2-lab1-supporting-services.adoc[Deploy Supporting Services] and that the PostgreSQL and Kafka pods are running in the `etx-app-dev` namespace.

== Run Quarkus in Development Mode

Open a terminal in the IDE and start Quarkus in dev mode:

[source,bash]
----
./mvnw compile quarkus:dev
----

Quarkus will connect to the external PostgreSQL and Kafka services deployed in the previous lab.
You should see log output confirming the connections:

----
Hibernate:
    create table if not exists ...
2026-05-09 20:08:28,533 INFO  [io.qua.sma.dep.processor] Configuring channel 'emails-in' to be managed by connector 'smallrye-kafka'
----

The application will be available on port 8080.
Click the endpoint notification in the IDE or find the URL in the *Endpoints* panel.

=== External Services Configuration

The application uses the following external services:

* **PostgreSQL** - `parasol-db.etx-app-dev.svc.cluster.local:5432`
  - Database: `parasol`
  - User/Password: `parasol/parasol` (from `parasol-db-secret`)

* **Kafka** - `parasol-kafka-kafka-bootstrap.etx-app-dev.svc.cluster.local:9092`
  - Topic: `intake`

These services are shared across all development workspaces, providing a consistent environment.

**Benefits of this approach:**

* ✅ Uses Red Hat certified container images only
* ✅ Consistent with production deployment patterns
* ✅ Avoids Podman rootless compatibility issues
* ✅ Shared services across team workspaces
* ✅ Easier to debug integration issues

IMPORTANT: Dev Services (Testcontainers) are **disabled** for this workshop. All services run externally in the cluster for consistency and reliability.

== Workspace Lifecycle

Some useful operations:

*Stop a workspace* (preserves state on the PVC):

* From the dashboard, click the stop icon next to your workspace
* The workspace pod is deleted, but the PVC retains your files

*Restart a workspace*:

* Click the workspace name in the dashboard to restart it
* Your source code, Maven cache, and other volume-mounted data persist

*Delete a workspace*:

* From the dashboard, select the workspace and click btn:[Delete]
* This removes the workspace pod and its PVC
