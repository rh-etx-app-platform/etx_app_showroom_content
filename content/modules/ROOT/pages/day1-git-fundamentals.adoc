= Git Fundamentals Lab
:navtitle: Git Fundamentals Lab

== Introduction

*Duration:* 30 minutes

In this lab, you will learn the fundamental Git operations that form the foundation for modern software development workflows. These skills are essential for working with CI/CD pipelines and GitOps practices in the ETX Application Platform.

=== Learning Objectives

By the end of this lab, you will be able to:

* Clone a Git repository and make local changes
* Stage and commit changes with meaningful messages
* Create and switch between branches
* Merge changes from one branch to another
* Push and pull changes to/from a remote repository
* Resolve basic merge conflicts

=== Prerequisites

* Access to GitLab instance: `{gitlab_url}`
* Command-line terminal access
* Basic understanding of command-line operations

== Lab Overview

This lab simulates a real-world development workflow:

1. Clone an existing application repository
2. Create a feature branch for your changes
3. Make and commit changes to your code
4. Merge your feature back to the main branch
5. Push your changes to the remote repository

== Exercise 2: Making Changes

=== Step 1: Create a Feature Branch

Create a new branch for your feature work:

[source,bash,role=execute]
----
git checkout -b feature/update-readme
----

.Understanding the Command
* `checkout -b` creates a new branch and switches to it
* Branch name follows convention: `feature/<description>`

Verify you're on the new branch:

[source,bash,role=execute]
----
git branch
----

You should see:
[source,text]
----
* feature/update-readme
  main
----

The asterisk (*) indicates your current branch.

=== Step 2: Make Changes to a File

Edit the README file to add your name:

[source,bash,role=execute]
----
cat >> README.md << 'EOF'

## Contributors
- Your Name ($(date +%Y-%m-%d))
EOF
----

=== Step 3: Stage and Commit Changes

View what changed:

[source,bash,role=execute]
----
git status
git diff README.md
----

Stage the changes:

[source,bash,role=execute]
----
git add README.md
----

Commit with a descriptive message:

[source,bash,role=execute]
----
git commit -m "docs: add contributor information to README"
----

.Commit Message Best Practices
[NOTE]
====
Good commit messages:

* Start with a type: `feat:`, `fix:`, `docs:`, `refactor:`
* Use imperative mood: "add feature" not "added feature"
* Keep the subject line under 50 characters
* Add detailed description if needed after a blank line

Example:
[source,text]
----
feat: add user authentication module

- Implement JWT token generation
- Add login/logout endpoints
- Include password hashing with bcrypt
----
====

=== Step 4: View Commit History

[source,bash,role=execute]
----
git log --oneline --graph --all
----

This shows a visual representation of your commit history across all branches.

== Exercise 3: Working with Multiple Branches

=== Step 1: Create Another Feature Branch

First, switch back to main:

[source,bash,role=execute]
----
git checkout main
----

Create a second feature branch:

[source,bash,role=execute]
----
git checkout -b feature/update-version
----

=== Step 2: Make Different Changes

Edit a different file (create one if needed):

[source,bash,role=execute]
----
echo "version: 1.1.0" > VERSION
git add VERSION
git commit -m "chore: bump version to 1.1.0"
----

=== Step 3: Visualize Your Branches

[source,bash,role=execute]
----
git log --oneline --graph --all --decorate
----

You should see something like:

[source,text]
----
* a3f8d92 (HEAD -> feature/update-version) chore: bump version to 1.1.0
| * b7e4c21 (feature/update-readme) docs: add contributor information to README
|/
* c9d2a45 (main) Initial commit
----

[TIP]
====
This visualization shows:

* `HEAD` points to your current branch
* Two feature branches diverged from `main`
* Each branch has its own commit history
====

== Exercise 4: Merging Branches

=== Step 1: Merge First Feature

Switch to main branch:

[source,bash,role=execute]
----
git checkout main
----

Merge the README update:

[source,bash,role=execute]
----
git merge feature/update-readme
----

.Expected Output
[source,text]
----
Updating c9d2a45..b7e4c21
Fast-forward
 README.md | 3 +++
 1 file changed, 3 insertions(+)
----

[NOTE]
====
*Fast-forward merge:* Git simply moves the `main` pointer forward because there were no other changes on `main`. This is the simplest type of merge.
====

=== Step 2: Merge Second Feature

Merge the version update:

[source,bash,role=execute]
----
git merge feature/update-version
----

This creates a merge commit because both branches modified the repository.

=== Step 3: Clean Up Feature Branches

After merging, delete the feature branches:

[source,bash,role=execute]
----
git branch -d feature/update-readme
git branch -d feature/update-version
----

Verify only main remains:

[source,bash,role=execute]
----
git branch
----

== Exercise 5: Working with Remote Repository

=== Step 1: View Remote Configuration

[source,bash,role=execute]
----
git remote -v
----

This shows your remote repository URL (called `origin`).

=== Step 2: Push Changes to Remote

Push your local changes to GitLab:

[source,bash,role=execute]
----
git push origin main
----

.What Happens
* `git push` sends your commits to the remote repository
* `origin` is the default name for your remote
* `main` is the branch you're pushing

=== Step 3: Create and Push a New Branch

Create a new feature branch:

[source,bash,role=execute]
----
git checkout -b feature/add-license
echo "MIT License" > LICENSE
git add LICENSE
git commit -m "docs: add MIT license"
----

Push the new branch to remote:

[source,bash,role=execute]
----
git push -u origin feature/add-license
----

.Understanding the -u Flag
[NOTE]
====
The `-u` flag sets up tracking:

* Links your local branch to the remote branch
* Allows `git pull` and `git push` without specifying the branch
* Only needed the first time you push a new branch
====

=== Step 4: Verify in GitLab UI

1. Go to `{gitlab_url}/software-factory/parasol-insurance-lab`
2. Click on "Branches"
3. You should see your `feature/add-license` branch

== Exercise 6: Simulating Merge Conflicts

Merge conflicts occur when two branches modify the same part of a file. Let's create and resolve one.

=== Step 1: Create Conflicting Changes

First, make a change on main:

[source,bash,role=execute]
----
git checkout main
echo "# Parasol Insurance Platform" > HEADER.md
git add HEADER.md
git commit -m "docs: add header file"
----

Create a branch from an earlier point:

[source,bash,role=execute]
----
git checkout HEAD~1
git checkout -b feature/update-header
echo "# Parasol Insurance Application" > HEADER.md
git add HEADER.md
git commit -m "docs: add different header"
----

=== Step 2: Attempt to Merge

Switch to main and try to merge:

[source,bash,role=execute]
----
git checkout main
git merge feature/update-header
----

.Conflict Output
[source,text]
----
Auto-merging HEADER.md
CONFLICT (add/add): Merge conflict in HEADER.md
Automatic merge failed; fix conflicts and then commit the result.
----

=== Step 3: View the Conflict

[source,bash,role=execute]
----
cat HEADER.md
----

You'll see conflict markers:

[source,text]
----
<<<<<<< HEAD
# Parasol Insurance Platform
=======
# Parasol Insurance Application
>>>>>>> feature/update-header
----

=== Step 4: Resolve the Conflict

Edit the file to keep both changes:

[source,bash,role=execute]
----
cat > HEADER.md << 'EOF'
# Parasol Insurance Platform
## Application Suite
EOF
----

Stage the resolved file:

[source,bash,role=execute]
----
git add HEADER.md
----

Complete the merge:

[source,bash,role=execute]
----
git commit -m "merge: resolve header conflict combining both descriptions"
----

[TIP]
====
*Conflict Resolution Best Practices:*

1. Understand both changes before resolving
2. Test the merged code
3. Remove ALL conflict markers (`<<<<<<<`, `=======`, `>>>>>>>`)
4. Commit with a clear message explaining the resolution
====

== Exercise 7: Pull Changes

=== Step 1: Simulate Remote Changes

In a real workflow, other developers push changes. We'll simulate this:

[source,bash,role=execute]
----
# Create a new branch to simulate a teammate's work
git checkout -b simulate-remote-change
echo "## Changelog\n- Version 1.1.0 released" >> README.md
git add README.md
git commit -m "docs: add changelog section"
git push origin simulate-remote-change
----

=== Step 2: Switch Context and Pull

[source,bash,role=execute]
----
# Switch to main (simulating you working on something else)
git checkout main

# Merge the "remote" changes (in real life, someone else would push to main)
git merge simulate-remote-change

# Push to remote
git push origin main
----

In a real scenario with multiple developers, you'd use:

[source,bash]
----
git pull origin main
----

This combines `git fetch` (download changes) + `git merge` (integrate changes).

== Key Concepts Summary

[cols="1,3",options="header"]
|===
|Concept |Description

|`git clone`
|Creates a local copy of a remote repository

|`git add`
|Stages changes for commit (adds to "staging area")

|`git commit`
|Saves staged changes to local repository history

|`git branch`
|Creates, lists, or deletes branches

|`git checkout`
|Switches branches or restores files

|`git merge`
|Combines changes from different branches

|`git push`
|Sends local commits to remote repository

|`git pull`
|Fetches and merges remote changes to local branch

|`git status`
|Shows working tree status and staged changes

|`git log`
|Displays commit history
|===

== Git Workflow for CI/CD and GitOps

Understanding these Git fundamentals is crucial for the ETX Application Platform:

*This workflow enables:*

1. **Continuous Integration (CI):** Every push triggers automated builds and tests
2. **GitOps:** Git commits drive infrastructure and application state
3. **Collaboration:** Branches and merge requests enable team workflows
4. **Traceability:** Every change is tracked with commits and merge history

== Quick Reference Card

[source,bash]
----
# Essential Git Commands
git clone <url>                    # Clone repository
git status                         # Check current state
git add <file>                     # Stage changes
git commit -m "message"            # Commit staged changes
git push origin <branch>           # Push to remote
git pull origin <branch>           # Pull from remote

# Branch Operations
git branch                         # List branches
git branch <name>                  # Create branch
git checkout <branch>              # Switch branch
git checkout -b <branch>           # Create and switch
git merge <branch>                 # Merge branch
git branch -d <branch>             # Delete branch

# Inspection
git log --oneline --graph --all    # Visual commit history
git diff                           # Show changes
git show <commit>                  # Show commit details

# Undo Changes
git restore <file>                 # Discard working changes
git reset HEAD <file>              # Unstage changes
git revert <commit>                # Revert commit
----
