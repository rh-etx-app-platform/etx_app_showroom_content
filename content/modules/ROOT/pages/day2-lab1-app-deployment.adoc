= Lab 1: Application Deployment with Argo CD
:navtitle: Application Deployment

== Overview

Deploy your own Argo CD instance to manage application deployments using a GitOps workflow.
The CI pipeline from the previous section pushes container images; Argo CD watches a GitOps repository and syncs Kubernetes manifests to the cluster.

IMPORTANT: You will deploy your **own** Argo CD instance in the `parasol-dev` namespace. The pre-deployed OpenShift GitOps operator in `openshift-gitops` is used for infrastructure automation only and is separate from your application deployment workflows.

== Install OpenShift GitOps Operator

The OpenShift GitOps operator is already installed cluster-wide from the infrastructure setup.
Verify it's available:

[source,bash,role=execute]
----
oc get csv -n openshift-operators | grep gitops
----

You should see `openshift-gitops-operator` with phase `Succeeded`.

== Deploy Your Argo CD Instance

Create an Argo CD instance in your namespace for managing application deployments:

[source,yaml]
----
apiVersion: argoproj.io/v1beta1
kind: ArgoCD
metadata:
  name: argocd
  namespace: parasol-dev
spec:
  server:
    autoscale:
      enabled: false
    grpc:
      ingress:
        enabled: false
    ingress:
      enabled: false
    resources:
      limits:
        cpu: 500m
        memory: 256Mi
      requests:
        cpu: 125m
        memory: 128Mi
    route:
      enabled: true
      tls:
        termination: edge
        insecureEdgeTerminationPolicy: Redirect
    service:
      type: ClusterIP
  grafana:
    enabled: false
    ingress:
      enabled: false
    route:
      enabled: false
  prometheus:
    enabled: false
    ingress:
      enabled: false
    route:
      enabled: false
  initialSSHKnownHosts: {}
  sso:
    provider: dex
    dex:
      openShiftOAuth: true
      resources:
        limits:
          cpu: 500m
          memory: 256Mi
        requests:
          cpu: 250m
          memory: 128Mi
  rbac:
    defaultPolicy: 'role:readonly'
    policy: |
      g, system:cluster-admins, role:admin
      g, cluster-admins, role:admin
    scopes: '[groups]'
  repo:
    resources:
      limits:
        cpu: 1000m
        memory: 1Gi
      requests:
        cpu: 250m
        memory: 256Mi
  resourceExclusions: |
    - apiGroups:
      - tekton.dev
      clusters:
      - '*'
      kinds:
      - TaskRun
      - PipelineRun
  controller:
    processors: {}
    resources:
      limits:
        cpu: 2000m
        memory: 2Gi
      requests:
        cpu: 250m
        memory: 1Gi
  ha:
    enabled: false
    resources:
      limits:
        cpu: 500m
        memory: 256Mi
      requests:
        cpu: 250m
        memory: 128Mi
----

Apply the ArgoCD CR:

[source,bash,role=execute]
----
oc apply -f argocd.yaml
----

Wait for Argo CD to be ready (this may take 3-5 minutes):

[source,bash,role=execute]
----
oc get pods -n parasol-dev -w
----

Press Ctrl+C when you see all `argocd-*` pods showing `1/1 Ready`.

=== Dev Environment Manifests

Create `environments/dev/namespace.yaml`:

[source,yaml]
----
apiVersion: v1
kind: Namespace
metadata:
  name: parasol-dev
----

Create `environments/dev/deployment.yaml`:

[source,yaml]
----
apiVersion: apps/v1
kind: Deployment
metadata:
  name: parasol-insurance
  namespace: parasol-dev
  labels:
    app: parasol-insurance
  annotations:
    instrumentation.opentelemetry.io/inject-java: "true" # <1>
spec:
  replicas: 1
  selector:
    matchLabels:
      app: parasol-insurance
  template:
    metadata:
      labels:
        app: parasol-insurance
      annotations:
        instrumentation.opentelemetry.io/inject-java: "true" # <1>
    spec:
      containers:
        - name: parasol-insurance
          image: image-registry.openshift-image-registry.svc:5000/parasol-dev/parasol-insurance:latest # <2>
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: 250m
              memory: 512Mi
            limits:
              cpu: "1"
              memory: 1Gi
----
<1> Enables OpenTelemetry auto-instrumentation. The Instrumentation CR created in the xref:day2-lab1-supporting-services.adoc[Supporting Services] section will inject the Java agent automatically.
<2> The CI pipeline updates this image tag after a successful build.

Create `environments/dev/service.yaml`:

[source,yaml]
----
apiVersion: v1
kind: Service
metadata:
  name: parasol-insurance
  namespace: parasol-dev
spec:
  selector:
    app: parasol-insurance
  ports:
    - port: 8080
      targetPort: 8080
----

Create `environments/dev/route.yaml`:

[source,yaml]
----
apiVersion: route.openshift.io/v1
kind: Route
metadata:
  name: parasol-insurance
  namespace: parasol-dev
spec:
  to:
    kind: Service
    name: parasol-insurance
  port:
    targetPort: 8080
  tls:
    termination: edge
----

Commit and push the dev environment:

[source,bash,role=execute]
----
git add environments/dev/
git commit -m "Add dev environment manifests"
git push origin main
----

== Create the Argo CD Application

Grant Argo CD permission to manage resources in the `parasol-dev` namespace:

[source,bash,role=execute]
----
oc adm policy add-role-to-user admin \
  system:serviceaccount:parasol-dev:argocd-application-controller \
  -n parasol-dev
----

Create an Argo CD Application that watches the `dev` environment.

NOTE: Replace `{gitlab_url}` in the YAML below with the actual GitLab URL from your environment before applying.

[source,yaml,subs="attributes+"]
----
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: parasol-dev
  namespace: parasol-dev
spec:
  project: default
  source:
    repoURL: {gitlab_url}/software-factory/parasol-gitops.git
    targetRevision: main
    path: environments/dev
  destination:
    server: https://kubernetes.default.svc
    namespace: parasol-dev
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=false # <1>
----
<1> Namespace already exists, so don't create it

Apply it:

[source,bash,role=execute]
----
oc apply -f parasol-dev-app.yaml
----

== Integrate CI and GitOps

The CI pipeline needs to update the image tag in the GitOps repository after a successful build.
Add a final task to the pipeline that commits the new image tag:

[source,yaml]
----
apiVersion: tekton.dev/v1
kind: Task
metadata:
  name: update-gitops-repo
  namespace: parasol-dev
spec:
  params:
    - name: gitops-repo-url
      type: string
    - name: image-tag
      type: string
    - name: environment
      type: string
      default: dev
  workspaces:
    - name: source
  steps:
    - name: update-image
      image: registry.access.redhat.com/ubi9/ubi:latest
      script: |
        #!/bin/bash
        set -e

        # Install git
        dnf install -y git && dnf clean all

        git clone $(params.gitops-repo-url) /tmp/gitops
        cd /tmp/gitops/environments/$(params.environment)

        # Update the image tag in the deployment manifest
        sed -i "s|image:.*parasol-insurance:.*|image: image-registry.openshift-image-registry.svc:5000/parasol-dev/parasol-insurance:$(params.image-tag)|" deployment.yaml

        git config user.email "pipeline@openshift.local"
        git config user.name "Tekton Pipeline"
        git add .
        git commit -m "Update parasol-insurance image to $(params.image-tag)"
        git push
----

Apply the task:

[source,bash,role=execute]
----
oc apply -f update-gitops-task.yaml
----

Update the pipeline to include this task at the end.

NOTE: Replace `{gitlab_url}` with your actual GitLab URL.

[source,yaml,subs="attributes+"]
----
    - name: update-gitops
      runAfter: [build-image]
      taskRef:
        name: update-gitops-repo
      params:
        - name: gitops-repo-url
          value: {gitlab_url}/software-factory/parasol-gitops.git
        - name: image-tag
          value: $(params.git-revision)
        - name: environment
          value: dev
      workspaces:
        - name: source
          workspace: shared-workspace
----

When the pipeline pushes the updated manifest, Argo CD detects the change and automatically syncs the new image to the cluster.

== Multi-Cluster Promotion to Stage

IMPORTANT: The secondary cluster for staging has been pre-configured as part of the bootstrap process. Argo CD already has the necessary credentials to deploy to the staging cluster.

[source,bash,role=execute]
----
# Login to Argo CD (if not already)
argocd login $(oc get route argocd-server -n parasol-dev -o jsonpath='{.spec.host}') \

Create stage environment manifests in your GitOps repository:

[source,bash,role=execute]

cd parasol-gitops
mkdir -p environments/stage

Copy the dev manifests as a template:

[source,bash,role=execute]

cp environments/dev/*.yaml environments/stage/

apiVersion: v1

  name: parasol-staging
----

Update `environments/stage/deployment.yaml` to use the `parasol-staging` namespace:

[source,yaml]
----
apiVersion: apps/v1
kind: Deployment
metadata:
  name: parasol-insurance
  namespace: parasol-staging # <1>
  labels:
    app: parasol-insurance
  annotations:
    instrumentation.opentelemetry.io/inject-java: "true"
spec:
  replicas: 2 # <2>
  selector:
    matchLabels:
      app: parasol-insurance
  template:
    metadata:
      labels:
        app: parasol-insurance
      annotations:
        instrumentation.opentelemetry.io/inject-java: "true"
    spec:
      containers:
        - name: parasol-insurance
          image: {quay_url}/parasol/parasol-insurance:stable # <3>
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: 500m
              memory: 1Gi
            limits:
              cpu: "2"
              memory: 2Gi
----
<1> Update namespace to parasol-staging
<2> Increase replicas for higher availability
<3> Use Quay registry for cross-cluster image distribution. Replace `{quay_url}` with your actual Quay URL.

Update `environments/stage/service.yaml` and `environments/stage/route.yaml` to use `parasol-staging` namespace.

Commit and push:

[source,bash,role=execute]
----
git add environments/stage/
git commit -m "Add stage environment manifests"
git push origin main
----

=== Create Stage Application

Create an Argo CD Application for the staging environment on the secondary cluster.

First, get the secondary cluster server URL from Argo CD:

[source,bash,role=execute]
----
# Get the staging cluster server URL
STAGING_CLUSTER=$(argocd cluster list -o json | jq -r '.[] | select(.name | contains("staging")) | .server')
echo "Staging cluster: ${STAGING_CLUSTER}"
----

Create the Application manifest:

NOTE: Replace `<STAGING_CLUSTER_URL>` with the URL from the command above, and `{gitlab_url}` with your GitLab URL.

[source,yaml,subs="attributes+"]
----
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: parasol-stage
  namespace: parasol-dev
spec:
  project: default
  source:
    repoURL: {gitlab_url}/software-factory/parasol-gitops.git
    targetRevision: main
    path: environments/stage
  destination:
    server: <STAGING_CLUSTER_URL> # <1>
    namespace: parasol-staging
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
----
<1> Points to the secondary cluster (pre-configured by bootstrap)

Apply it:

[source,bash,role=execute]
----
oc apply -f parasol-stage-app.yaml
----

== Promotion Workflow Overview

The promotion workflow from dev to staging follows these steps:

. CI pipeline builds and pushes image to internal registry with commit SHA tag
. Image is promoted to Quay registry with a `stable` tag (covered in xref:day2-lab1-promotion.adoc[Promotion Pipeline])
. GitOps repository `environments/stage/deployment.yaml` is updated with new image tag
. Argo CD detects the change and automatically syncs to the secondary cluster

This GitOps approach ensures:

* **Version control** - All deployment changes are tracked in Git
* **Auditability** - Full history of what was deployed when
* **Rollback capability** - Revert to previous commit to rollback
* **Multi-cluster consistency** - Same manifests can deploy to multiple clusters

== Quick Verification

Verify the complete deployment:

[source,bash]
----
# Check Argo CD application status
oc get application parasol-dev -n parasol-dev -o jsonpath='{.status.sync.status}: {.status.health.status}'

# Check application pods
oc get pods -n parasol-dev -l app.kubernetes.io/name=parasol-app

# Check application health endpoint
curl http://$(oc get route parasol-app -n parasol-dev -o jsonpath='{.spec.host}')/q/health

# Check OpenTelemetry injection
oc get pod -n parasol-dev -l app.kubernetes.io/name=parasol-app \
  -o jsonpath='{.items[0].spec.initContainers[*].name}'
----

Expected output should show: `Synced: Healthy` for Argo CD, `Running` pods, and `opentelemetry-auto-instrumentation` init container.

[TIP]
====
**Common Issues:**

* **ImagePullBackOff:** Image not in registry - run CI pipeline first
* **CrashLoopBackOff:** Check pod logs - often missing Kafka or DB connection
* **Argo CD Out of Sync:** Click "Sync" in Argo CD UI or check GitOps repo for errors
* **Health check failing:** Verify all dependencies (DB, Kafka) are running
====
