= GitLab Integration and Shared Secrets
:navtitle: GitLab Integration & Shared Secrets

== GitLab OAuth for SCM Integration

Connecting Dev Spaces to GitLab allows automatic Git credential injection into workspaces -- developers can clone, push, and pull without manually configuring tokens.

=== Create a GitLab OAuth Application

. Open GitLab and authenticate with your SSO credentials.
. Click your avatar in the upper-right corner.
. Select menu:Edit profile[Applications] to open the User Settings applications page.
+
[source,text,subs="attributes+"]
----
{gitlab_url}/-/user_settings/applications
----
. Click btn:[Add new application].
. Create the application with the following settings:
+
[cols="1,2"]
|===
| Name | `devspaces`
| Redirect URI | `pass:a[https://devspaces.{openshift_cluster_ingress_domain}/api/oauth/callback]`
| Confidential | Checked
| Scopes | `api`, `write_repository`, `openid`
|===
. Click btn:[Save application].
. Note the *Application ID* and *Secret*

=== Create the OAuth Secret

Create a Kubernetes secret that Dev Spaces will use to authenticate against GitLab.
The labels and annotations are required -- Dev Spaces discovers the secret automatically based on them.

[source,bash,subs="attributes"]
----
GITLAB_APPLICATION_ID="REPLACE_APPLICATION_ID"
GITLAB_APPLICATION_SECRET="REPLACE_APPLICATION_SECRET"

oc apply -f - <<EOF
apiVersion: v1
kind: Secret
metadata:
  name: gitlab-oauth-config
  namespace: openshift-devspaces
  labels:
    app.kubernetes.io/part-of: che.eclipse.org
    app.kubernetes.io/component: oauth-scm-configuration
  annotations:
    che.eclipse.org/oauth-scm-server: gitlab
    che.eclipse.org/scm-server-endpoint: {gitlab_url}
type: Opaque
stringData:
  id: "$GITLAB_APPLICATION_ID"
  secret: "$GITLAB_APPLICATION_SECRET"
EOF
----

WARNING: Replace `GITLAB_APPLICATION_ID` and `GITLAB_APPLICATION_SECRET` before running the command.

=== Register GitLab in the CheCluster

Add the `gitServices` section to your CheCluster CR:

[source,bash,subs="attributes"]
----
oc patch checluster/devspaces -n openshift-devspaces \
  --type='merge' \
  -p '{"spec":{"gitServices":{"gitlab":[{"secretName":"gitlab-oauth-config","endpoint":"{gitlab_url}"}]}}}'
----

When a user opens a workspace that references a GitLab repository, they will be prompted to authenticate with GitLab via OAuth.
This grants push/pull access using their personal GitLab credentials.

== Mounting Shared Secrets into Workspaces

Dev Spaces can automatically mount Kubernetes Secrets (and ConfigMaps) into every workspace pod using labels and annotations.
This is useful for providing API keys, service credentials, or shared configuration to all developers without them having to configure it manually.

=== How It Works

Any Secret or ConfigMap in the `openshift-devspaces` namespace with the label `app.kubernetes.io/part-of: che.eclipse.org` and the annotation `controller.devfile.io/mount-as` will be automatically mounted into workspace containers.

Mount options:

* `mount-as: env` -- Inject as environment variables
* `mount-as: file` -- Mount as files at a specified path
* `mount-as: subpath` -- Mount individual keys as separate files

=== Example: AI Code Assistant (Continue) Credentials

This example creates a secret that injects LLM API credentials as environment variables into every workspace. These can be provided to an extension such as Roo, Cline, or Continue to enable LLM-assisted coding.

First, obtain an API key from the https://litellm-prod-frontend.apps.maas.redhatworkshops.io/home[Red Hat Demo Platform MaaS^].
Log in with your SSO credentials. If you do not have access to this service, ask the instructor for the workshop-provided LLM endpoint and API key.

If you don't have an API key, you can create one by following these steps:

. Navigate to *Models* and find *qwen3-14b*
. Click *Subscribe* to subscribe to the model
. If the model already shows an active subscription, continue to the API key step
. Go to *API Keys* and click *Create API Key*
. Provide a name for the key and select the *qwen3-14b* subscription
. Save the generated key somewhere safe

If you already have an API key, you can use it by following these steps:

. Go to *API Keys* and select the API Key subscribed to the *qwen3-14b* model
. Click the *show key* button to view the API key
. Copy the generated key

Then create the secret:

[source,bash,subs="attributes"]
----
LLM_API_KEY="REPLACE_API_KEY"

oc apply -f - <<EOF
apiVersion: v1
kind: Secret
metadata:
  name: continue-llm-credentials
  namespace: openshift-devspaces
  labels:
    app.kubernetes.io/part-of: che.eclipse.org
    app.kubernetes.io/component: workspaces-config
  annotations:
    controller.devfile.io/mount-as: env
type: Opaque
stringData:
  LLM_API_KEY: "$LLM_API_KEY"
  LLM_BASE_URL: https://litellm-prod.apps.maas.redhatworkshops.io
EOF
----

WARNING: Replace `LLM_API_KEY` before running the command.

Every new workspace will now have `LLM_API_KEY` and `LLM_BASE_URL` available as environment variables.
The devfile's `postStart` command can use these to configure the Continue extension automatically (see xref:day2-devspaces-devfiles.adoc[Devfiles]).
